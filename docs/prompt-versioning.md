# Agent Contract Versioning

Agent Contracts evolve. Projects running in production depend on specific
contract behavior and must not break when upstream contracts change. This
document defines the versioning convention for all Agent Contracts in
CodeConductor.

---

## Why Version Agent Contracts

An Agent Contract is not a suggestion — it is a behavioral specification that
determines how a Conductor Agent interprets tasks, makes decisions, and produces
Deliverables. When a contract changes, agent behavior changes. Projects that
depend on a specific behavior need reproducibility.

Versioning ensures:

- Projects can pin to a known-good contract version
- New improvements can ship without breaking existing projects
- The history of why each contract changed is preserved
- Rollback is always possible — no version is ever deleted

---

## Single Source of Truth

The canonical version is the `version` field in `package.json`.

```jsonc
// package.json
{
  "version": "0.1.0",
}
```

Agent Contract paths and the `contract_version` field in
`codeconductor.config.jsonc` must always match this value at release time.

---

## Path Structure

Agent Contracts live under the preset directory, organized by version:

```text
presets/
  opencode/
    prompts/
      v0.1.0/
        orchestrator.md
        task-coach.md
        architect.md
        implementer.md
        tester.md
        reviewer.md
        docs.md
        repo-explorer.md
      v0.2.0/           # unreleased draft — not active until package.json changes
        orchestrator.md
        ...
    agents/             # symlink-style reference — always points to active version
      orchestrator.md
      task-coach.md
      ...
```

The `agents/` directory contains the contracts that OpenCode loads directly. The
`prompts/{version}/` directories are the versioned archive.

When CodeConductor scaffolds a project, it copies the resolved version of each
contract into the target project and records the version in config.

## How to Update a Contract

Follow these steps in order. Do not skip steps.

### 1. Create the new version directory

```bash
cp -r presets/opencode/prompts/v0.1.0 presets/opencode/prompts/v0.2.0
```

Always copy from the previous version. Never start from scratch — contracts
accumulate intent that is easy to lose.

### 2. Edit only what needs to change

Open the specific contract file(s) in the new directory. Change only what is
necessary. Unrelated contracts must be identical to the previous version.

### 3. Update the version table in this file

Add a row to the version table below with the new version, the date, a summary
of what changed, and mark the previous version as `deprecated`.

### 4. Document the change in CHANGELOG.md

Under `[Unreleased]` or the appropriate release section, add a `Changed` entry
describing:

- Which contract changed
- What the behavioral difference is
- Why the change was made

### 5. Update `agents/` to reflect the new active version

Copy or overwrite the affected files in `agents/` with the new version content.

```bash
cp presets/opencode/prompts/v0.2.0/orchestrator.md presets/opencode/agents/orchestrator.md
```

### 6. Never delete previous versions

Projects in production reference specific versions. Deleting them breaks those
projects. Mark versions as `deprecated` in the table — that is the extent of
retirement.

## How a Project References a Contract Version

When CodeConductor initializes a target project, it generates a
`codeconductor.config.jsonc` at the project root:

```jsonc
{
  // CodeConductor configuration
  // Generated by: npx codeconductor init
  "target": "opencode",
  "stack": "spring-boot-kotlin",

  // Pin to the exact contract version used during init.
  // Update this field explicitly when you want to adopt new contract behavior.
  "contract_version": "0.1.0",

  "agents": {
    "orchestrator": true,
    "task-coach": true,
    "architect": true,
    "implementer": true,
    "tester": true,
    "reviewer": true,
    "docs": true,
    "repo-explorer": true,
  },
}
```

The `contract_version` field is intentionally explicit. Upgrading to a new
contract version is a deliberate act, not an automatic update.

## Naming Convention

Versions follow the same semver as `package.json`: `v{major}.{minor}.{patch}`.

| Bump    | When to use                                                          |
| ------- | -------------------------------------------------------------------- |
| `patch` | Typo fixes, clarifications that do not change agent behavior         |
| `minor` | New instructions, expanded scope, or new output fields               |
| `major` | Breaking behavioral change — output format or decision logic changes |

---

## Core Rule

**Never break an active contract. Deprecate, do not delete.**

A project that pinned to `v0.1.0` must be able to reinstall from that version
indefinitely. If a new version changes behavior incompatibly, the old version
stays in the repository and the new version ships alongside it.
